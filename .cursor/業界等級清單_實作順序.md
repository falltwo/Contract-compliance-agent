# 業界等級清單與實作順序

可勾選、可排優先級。建議依序：**1 → 2 → 3**，4 視需求再做。

---

## 1️⃣ Evaluation Framework

**目標**：能回答「accuracy 幾 %？routing 準確率？latency？tool 成功率？」

| 優先 | 項目 | 說明 | 狀態 |
|------|------|------|------|
| P0 | **Eval 題集** | 建立 `eval_set.json`（或 CSV）：20～50 題，每題含 `question`、可選 `expected_tool`、`expected_sources` 或簡短 `expected_answer` | ☑ |
| P0 | **批次跑 Eval** | 腳本讀題集 → 對每題呼叫 `route_and_answer` → 寫入結果（predicted_tool, answer, latency, source_count） | ☑ |
| P0 | **核心指標計算** | 從結果算：routing 準確率（predicted_tool == expected_tool）、latency P50/P95、tool call 成功率（無 exception）、可選 answer 長度/是否空 | ☑ |
| P1 | **Eval Dashboard 指標頁** | Streamlit 或獨立頁：顯示「本次 eval run」的 accuracy %、latency 分布、tool 混淆矩陣（可接 Phase 2） | ☐ |
| P1 | **Hallucination / 品質指標** | 可選：LLM-as-judge 對「答案是否依據檢索」打分，或人工抽樣標註 → 算 hallucination rate | ☐ |

**產出**：跑一次 eval 就有「routing 準確率 X%、latency P95 Y 秒」等數字，改 prompt/model 可前後對比。

**如何跑 Eval**（專案根目錄）：`uv run python eval/run_eval.py`；可選 `--groq`（改用 Groq，需設 `GROQ_API_KEY`）、`--eval-set`、`--out-dir`。結果寫入 `eval/runs/run_<timestamp>_results.jsonl`，指標寫入 `run_<timestamp>_metrics.json`。  
**注意**：請勿在 `.env` 設定 `EVAL_USE_GROQ=1`，否則 Streamlit 對話也會用 Groq；僅在跑 Eval 時加 `--groq` 即可讓「只有該次 eval」使用 Groq。

## 2️⃣ Logging / Observability

**目標**：tracing、tool call logs、token/cost、failure 可追蹤。

| 優先 | 項目 | 說明 | 狀態 |
|------|------|------|------|
| P0 | **Request ID / Trace ID** | 每筆請求進入時產生 `request_id`（uuid），在 log 與 eval 記錄中帶上，方便串起同一輪對話的所有 log | ☐ |
| P0 | **結構化 Tool Call Log** | 每次執行 tool 前後寫一筆：request_id, tool_name, input_summary, latency_sec, success(bool), error_message(若失敗) | ☐ |
| P1 | **Token Usage** | 若 Gemini API 回傳 token 數，在 log / eval 記錄中寫入 prompt_tokens, completion_tokens，方便算 cost | ☐ |
| P1 | **Cost Tracking** | 依 token 與單價算出每筆請求成本，可寫入 DB 或 JSONL，dashboard 可顯示日/週總成本 | ☐ |
| P2 | **Failure Retry 記錄** | 若有 retry（見 3️⃣），每次 retry 寫一筆 log（request_id, tool, attempt, result） | ☐ |

**產出**：用 request_id 可還原「這一問用了哪些 tool、各花幾秒、有沒有錯、用了多少 token」。

## 3️⃣ Error Handling Design

**目標**：timeout、tool 失敗、partial answer 有明確定義與 fallback。

| 優先 | 項目 | 說明 | 狀態 |
|------|------|------|------|
| P0 | **Tool Timeout** | 對外呼叫（Tavily、Firecrawl、Gemini）設 timeout（e.g. 30s），逾時捕獲並回傳明確錯誤訊息給使用者，並 log | ☐ |
| P0 | **Web Tool Fail Fallback** | 政策寫死：若 `web_search` 失敗（timeout/API error）→ 可選自動改試 `firecrawl_search` 或直接回「網路搜尋暫時無法使用，請稍後再試」並 log | ☐ |
| P0 | **Partial Answer / 空答案** | 若 LLM 回傳空或只回「抱歉」：log、可選 retry 一次 generate，或回固定提示「無法產生回答，請換個問法」 | ☐ |
| P1 | **Retry Policy** | 對可重試錯誤（如 5xx、rate limit）訂次數（e.g. 2 次）與 backoff，寫在共用的「呼叫外部 API」包裝裡 | ☐ |
| P1 | **Tool Hallucination 緩解** | 若 routing 選錯 tool（可事後用 eval 看出）：可加「tool 回傳空或錯誤時，是否允許 router 再選一次」的開關或策略 | ☐ |

**產出**：有一份簡短「Error Handling 政策」文件（timeout 幾秒、誰 fallback 到誰、retry 幾次），程式行為與 log 符合該政策。

## 4️⃣ Scalable Design（視需求再做）

**目標**：從單機 demo 走向可擴展、可併發、可部署。

| 優先 | 項目 | 說明 | 狀態 |
|------|------|------|------|
| P2 | **Async 呼叫** | 對可並行的外部呼叫（如多個 tool、或 retrieve + 其他）改為 async，減少單請求總延遲 | ☐ |
| P2 | **Caching** | 對相同 question 的 RAG 檢索結果或 embedding 做 cache（記憶體或 Redis），減少重複呼叫 | ☐ |
| P2 | **API 化** | 將「問答」拆成獨立 API（FastAPI/Flask），Streamlit 只當前端呼叫 API，方便之後換成其他前端或加 queue | ☐ |
| P3 | **Queue 系統** | 高負載時請求先進 queue（Redis/RabbitMQ），worker 依序處理，避免單機爆掉 | ☐ |
| P3 | **Distributed Tool Server** | 將部分 tool（如 Firecrawl、圖表）拆成獨立服務，Agent 透過 HTTP/gRPC 呼叫，方便擴展與限流 | ☐ |

**產出**：架構圖與部署方式可支援「多實例、有 cache、有 queue」，而不是單一 Streamlit process。

---

## 建議實作順序（第一輪）

1. **1️⃣ P0**：Eval 題集 + 批次跑 + 核心指標（routing 準確率、latency、tool 成功率）→ 先有數字。
2. **2️⃣ P0**：Request ID + 結構化 Tool Call Log → 先可追蹤。
3. **3️⃣ P0**：Tool Timeout + Web Fail Fallback + Partial Answer 處理 → 先有錯誤政策。

完成以上，就可以說「有 evaluation 數字、有 observability、有 error policy」；再視需要補 2️⃣ P1（token/cost）、1️⃣ P1（dashboard/LLM judge）、3️⃣ P1（retry），最後才動 4️⃣。
